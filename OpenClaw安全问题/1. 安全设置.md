# 第 0 步：先更新系统并装基础工具

Ubuntu 的安全更新默认通过 `unattended-upgrades` 自动处理，默认每天运行一次；但在你开始部署前，先把系统更新到最新仍然是必要的。

```
sudo apt update && apt full-upgrade -y
sudo apt install -y curl git vim ufw fail2ban unattended-upgrades ca-certificates
reboot
```

重连后继续。

# 第 1 步：先建一个普通管理员用户，并给它设置 sudo 密码

## 1. 如果是直接root显示登陆的需要做！

在 Ubuntu 里，`sudo` 默认让你输入的是**当前用户自己的密码**。
 所以正确做法不是先给 root 搞密码，而是：

- 创建一个普通用户
- 把它加入 `sudo` 组
- 给这个用户设密码
- 以后用这个用户 + `sudo`

### 具体推荐这样做

下面我用 `ubuntu` 举例，你可以换成你想要的用户名。

```
adduser ubuntu
usermod -aG sudo ubuntu
passwd
```

然后把你当前 root 账户里的 SSH 公钥复制给这个用户：

```
rsync -a --chown=ubuntu:ubuntu /root/.ssh /home/ubuntu/
chmod 700 /home/ubuntu/.ssh
chmod 600 /home/ubuntu/.ssh/authorized_keys
```

现在**不要退出当前 root 会话**，新开一个终端测试：

```
ssh ubuntu@你的公网IP
```

# 第 2 步：立刻做 SSH 加固

## 1. 如果还没有SSH登陆， 先从密码到ssh登陆

```
ssh-keygen -t ed25519 -f C:\Users\Helios\.ssh\my_new_vps_ed25519 -C "Helios-Win-PC"
```

如果你用记事本打开它，你会发现它的格式是这样的：

> ```
> ssh-ed25519 AAAAC3NzaC1lZDI1NTE...[这里是一大长串毫无规律的乱码]... "Helios-Win-PC"
> ```

双引号里的这段备注，会直接跟在那串长长的乱码最后面。

公钥也要放进去：

```
ubuntu@vps-2a3a4c57:~/.ssh$ nano authorized_keys 
ubuntu@vps-2a3a4c57:~/.ssh$ chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

## 2. 然后给sudo加密：

### 解决“免密 sudo”的安全隐患

确认 SSH 密钥登录没问题后，我们来处理你最关心的 `sudo` 密码问题。在刚刚通过 `ssh ovh-us` 登录成功的终端里操作：

**1. 先给你的普通用户设置一个密码**（因为很多云厂商默认不给密码，如果不设密码直接取消免密 sudo，你就永远失去 root 权限了）：

Bash

```
sudo passwd $USER
```

*按提示输入并确认你的新密码，务必牢记！*

**2. 找出云厂商写死免密配置的文件：**

Bash

```
sudo grep -r NOPASSWD /etc/sudoers /etc/sudoers.d/
```

*它通常会输出类似 `/etc/sudoers.d/90-cloud-init-users:ubuntu ALL=(ALL) NOPASSWD:ALL` 的结果。*

**3. 删掉那个危险的 `NOPASSWD:`：** 假设上一步找到的文件是 `/etc/sudoers.d/90-cloud-init-users`，执行：

Bash

```
sudo visudo -f /etc/sudoers.d/90-cloud-init-users
```

将里面的 `NOPASSWD:` 删掉。例如把 `ubuntu ALL=(ALL) NOPASSWD:ALL` 改成 `ubuntu ALL=(ALL) ALL`。保存并退出（按 `Ctrl+O` 回车保存，`Ctrl+X` 退出）。

**4. 设置每次 sudo 强制要求密码：**

Bash

```
sudo visudo
```

在打开的文件中，找到带有 `Defaults` 的那些行，在它们下方新起一行加上：

表示0分钟每次都要输入密码

```
Defaults timestamp_timeout=0
```

保存并退出。

## 3. 路径和文件

OpenSSH 文档里，`PasswordAuthentication` 默认是 `yes`，`PubkeyAuthentication` 默认是 `yes`，SSH 端口默认是 `22`。你既然已经用 SSH key 登录了，就应当把密码登录关掉，并最终禁止 root 远程登录。

是该路径下和该文件

```
/etc/ssh/sshd_config
```

```
PermitRootLogin no
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
UsePAM yes
```

然后检查配置有没有语法错误：

```
sshd -t
```

如果没报错，再重载 SSH：

```
systemctl reload ssh
```

## 重中之重：这一步一定这样验证

1. **当前 root 会话不要关**
2. 新开一个窗口
3. 用 `ubuntu` 再 SSH 一次测试能否登录
4. 测试成功后，才算这一步完成



## 如果不成功：

哈哈，你刚好踩中了我在上一条回复里特别提醒你的 **“Ubuntu 隐藏陷阱”**！

这绝对不是你改错了（你贴出来的配置文件改得非常完美，前面的 `#` 也去掉了），原因主要出在以下两个最容易被忽略的地方。请依次排查：

### 嫌疑人 1号：潜伏的云厂商配置（优先级比你高）

仔细看你刚才发出来的配置文件，第 12 行有一句： `Include /etc/ssh/sshd_config.d/*.conf`

**SSH 配置文件的铁律是：“先入为主，谁在前面谁说了算”。** 因为这行 `Include`（包含命令）写在文件的最开头，所以 SSH 服务启动时，会**最优先**去读取 `/etc/ssh/sshd_config.d/` 文件夹里的配置。

很多云厂商（比如 OVH、AWS 等）会在这个文件夹里塞一个叫 `50-cloud-init.conf` 的文件，里面赫然写着 `PasswordAuthentication yes`。因为这行代码被系统最先读到，所以你写在主文件后面的 `no` 直接被系统无视了！

# 第 3 步：装 Tailscale，并先验证 Tailscale SSH 路径

Tailscale 官方 Linux 安装文档给出的主流发行版安装方式是：

```
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
```

完成后会给你一个浏览器授权链接；授权成功后，机器会出现在你的 tailnet 里。官方也建议服务器场景考虑关闭 key expiry，避免以后远程被锁在门外。

## 安装与接入

```
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up
```

然后查看这台 VPS 的 Tailscale IP：

```
tailscale ip -4
```

你会得到一个 `100.x.y.z` 的地址。

## 从你本地另一台已经登录 Tailscale 的设备测试

```
ssh david@100.x.y.z
```

**先确认这条路能通**，再去收公网 SSH。

## 很关键：关掉这台服务器的 Tailscale key expiry

Tailscale 文档说明可以在管理后台 Machines 页面把某台设备的 key expiry 关闭；如果 key 过期，连接会停止工作。对于服务器，关掉 expiry 通常更稳。

另外，官方也提醒：如果设备已经过期，不要只靠远程 Tailscale SSH 去做 `tailscale up --force-reauth`，因为这可能把你当前远程连接带断。

# 4. UFW防火墙

## 我建议你直接执行的命令

先确认接口名没看错：

```
ip -br a
```

确认公网网卡确实还是 `ens3`，Tailscale 还是 `tailscale0`。
 然后按下面执行。

## 你现在这台机的网卡名不是 `ens3`

你当前看到的是：

```
lo
eth0
eth1
```

所以你之前那套：

```
22/tcp on tailscale0
22/tcp on ens3
```

在这台新 VPS 上**不能直接照搬**，因为：

- **没有 `ens3`**
- **目前也还没有 `tailscale0`**

------

## 先把你当前网络状态解释清楚

你现在的输出是：

```
lo               UNKNOWN        127.0.0.1/8 ::1/128 
eth0             UP             170.64.176.61/20 10.49.0.5/16 fe80::...
eth1             UP             10.126.0.2/20 fe80::...
```

## 这代表什么

### `lo`

本机回环接口，不用管。

### `eth0`

这是你当前最关键的主网卡。
 因为它带了公网 IPv4：

```
170.64.176.61
```

所以你现在公网访问这台机器，主要就是走 **`eth0`**。

### `eth1`

这是另一块私网接口，只有私网地址：

```
10.126.0.2
```

通常用于私有网络、VPC、内网通信之类。

### 一次性重建 UFW（推荐方案）

> 目标：
>
> - 开放 `80/443`（网站）
> - SSH `22` **只允许 Tailscale**
> - 明确拒绝公网网卡 `ens3` 的 `22`
> - 明确拒绝 `18789/18792/5353`
> - 同时保留 IPv4 + IPv6 规则

------

### 0）先确认公网网卡名（你这里看起来是 ens3）

先跑一下，确认没写错：

```
ip route get 1.1.1.1 | awk '{print $5; exit}'
```

如果输出是 `ens3`，下面命令可以直接用。
 （如果不是，就把命令里的 `ens3` 换成你实际网卡名）

------

### 1）备份一下当前 UFW 配置（可选但建议）

```
sudo cp /etc/ufw/user.rules /etc/ufw/user.rules.bak.$(date +%F-%H%M%S) 2>/dev/null || true
sudo cp /etc/ufw/user6.rules /etc/ufw/user6.rules.bak.$(date +%F-%H%M%S) 2>/dev/null || true
```

------

### 2）彻底重置 UFW（清空所有规则）

```
sudo ufw --force disable
sudo ufw --force reset
```

------

### 3）确保 UFW 开启 IPv6（这样同一条命令会生成 v4+v6）

> 你刚才出现 “只有 v6” 的情况，多半是规则状态当时不完整或重复导致看起来怪。
> 正常情况下只要 `IPV6=yes`，同一条 `ufw allow/deny` 会同时生成 IPv4 和 IPv6 规则。

```
sudo sed -i 's/^IPV6=.*/IPV6=yes/' /etc/default/ufw
grep '^IPV6=' /etc/default/ufw
```

你应该看到：

```
IPV6=yes
```

------

### 4）设置默认策略（推荐）

```
sudo ufw default deny incoming
sudo ufw default allow outgoing
```

------

### 5）按顺序添加你需要的规则（这次是干净版，不会重复）

#### A. 网站端口（公网开放）

```
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
```

#### B. SSH 只走 Tailscale（重点）

```
sudo ufw allow in on tailscale0 to any port 22 proto tcp
sudo ufw deny in on eth0 to any port 22 proto tcp
```

#### C. OpenClaw/局域网发现相关端口（显式拒绝，加固）

```
sudo ufw deny 18789/tcp
sudo ufw deny 18792/tcp
sudo ufw deny 5353/udp
```

------

### 6）启用 UFW

```
sudo ufw --force enable
```

------

### 7）检查规则（看是否同时有 v4 + v6）

```
sudo ufw status numbered
sudo ufw status verbose
```

# F. 配 Fail2ban

```
sudo tee /etc/fail2ban/jail.d/sshd.local > /dev/null <<'EOF'
[DEFAULT]
bantime = 1h
findtime = 10m
maxretry = 5
backend = systemd
banaction = ufw
ignoreip = 127.0.0.1/8 ::1 100.64.0.0/10

[sshd]
enabled = true
EOF

sudo systemctl enable --now fail2ban
sudo systemctl restart fail2ban
sudo fail2ban-client status sshd
```

这里的核心意思是：

- 本机回环不封
- Tailscale CGNAT 网段 `100.64.0.0/10` 不封



## 你这段配置本身为什么能用

这几项都是 Fail2ban jail 的标准参数：

- `bantime`：封禁时长
- `findtime`：统计失败次数的时间窗
- `maxretry`：达到多少次失败后触发封禁
- `backend`：日志后端；Ubuntu 的手册里说明可用 `systemd`
- `banaction`：封禁动作；可以在 `[DEFAULT]` 里统一指定
   这些都在 Ubuntu 的 Fail2ban `jail.conf` 手册里有定义。



### 1. 查看“所有成功的登录记录”（推荐）

这个命令会读取系统的 `/var/log/wtmp` 文件，列出近期所有成功登入过系统的用户、IP 地址以及登录时间。

Bash

```
last
```

- **怎么看：** 第一列是用户名（比如 `ubuntu` 或 `root`）。 第三列是登录来源的 IP。如果你看到的是你的本地公网 IP 或者 Tailscale 的 `100.x.x.x` IP，那就非常安全。 *（如果记录太多，你可以按 `q` 退出，或者输入 `last -n 10` 只看最近 10 条）。*

### 2. 查看“所有失败的爆破记录”（看黑客碰壁）

这个命令读取的是 `/var/log/btmp` 文件，专门记录那些试图登录但密码或密钥错误的坏蛋。因为它涉及安全隐私，所以**需要加 `sudo`**：

Bash

```
sudo lastb
```

- **怎么看：** 你会看到一堆稀奇古怪的用户名（比如 `admin`、`test`、`root`）和各种不知名的外国 IP。这些就是之前被 Fail2ban 捕捉到的那 80 多次爆破尝试。看着他们全部失败，会有一种很爽的安全感！

```
last -ai
```

这个命令看的就是**成功登录历史**。所以这份记录里，除了 `reboot` 和 `wtmp begins` 以外，其他 `ubuntu` / `root` 开头的行，都是**系统记录到的成功会话**。

## 如果你是在自己的 Win / Mac 本机运行 OpenClaw

**不是“只需要装 Tailscale”**，而是：

1. **本机只用 localhost / 127.0.0.1 跑 OpenClaw** 不要暴露公网
2. **保持 OpenClaw 默认的 token auth**
3. **不要关闭 device auth / pairing**
4. **需要跨设备远程访问时，再装 Tailscale**

也就是说：

> **Tailscale 不是本地单机运行的必需品，而是“你想从别的设备安全访问这台机器上的 OpenClaw”时特别推荐的方案。**
>  本机自己用，哪怕不装 Tailscale，也可以是安全的。
