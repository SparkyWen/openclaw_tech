## 3. 手动添加 NodeSource 的 GPG key 和 repo

```
sudo mkdir -p /etc/apt/keyrings

curl -fsSL https://deb.nodesource.com/gpgkey/nodesource-repo.gpg.key \
  | sudo gpg --dearmor -o /etc/apt/keyrings/nodesource.gpg

echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/nodesource.gpg] https://deb.nodesource.com/node_24.x nodistro main" \
  | sudo tee /etc/apt/sources.list.d/nodesource.list > /dev/null
```

这里的关键点有两个：

- NodeSource 现在 DEB 包走的是 **`nodistro`** 路径。
- OpenClaw 官方给 Ubuntu / Debian 的 Node 安装示例，明确就是 **Node 24**。

# 2. 安装流程

**按 OpenClaw 官方当前文档，你这个一键命令会自动检测并安装 Node，不需要你先单独装 Node**。官方安装页写得很明确：`install.sh` 会在一条命令里处理 **Node 检测、Node 安装、OpenClaw 安装，以及 onboarding**；官网首页也写了 “**The one-liner installs Node.js and everything else for you**”。另外，官方 Node 文档说明 OpenClaw 需要 **Node 22 或更高版本**，而安装脚本会自动处理这一点。

```
curl -fsSL https://openclaw.ai/install.sh | bash
```

如果报错：

```
[3/3] Finalizing setup

! PATH missing npm global bin dir: /home/ubuntu/.npm-global/bin
  This can make openclaw show as "command not found" in new terminals.
  Fix (zsh: ~/.zshrc, bash: ~/.bashrc):
    export PATH="/home/ubuntu/.npm-global/bin:$PATH"
```

这和 OpenClaw 官方文档里的说明一致：`openclaw: command not found` 几乎总是因为 **npm global bin directory 不在 PATH 里**。安装器内部文档也明确说了，这类情况通常是 Linux 上把 npm 全局前缀切到了 `~/.npm-global`，但 shell 启动文件还没成功补上 PATH。

## 为什么它现在还能继续运行

因为安装脚本在当前安装会话里，已经知道 OpenClaw 的位置，所以它还能继续启动 onboarding。
 但这**不代表以后每个新 shell 会话都能直接找到 `openclaw` 命令**。官方安装器文档和 Node 安装文档都把这归类为 **PATH 配置问题，不是包没装上**。

## 你现在应该立刻修

你是 bash，就执行：

```
echo 'export PATH="$HOME/.npm-global/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
hash -r
```

然后验证：

```
which openclaw
openclaw --version
npm prefix -g
echo "$PATH"
```

OpenClaw 官方文档给出的 Linux/macOS 处理方式也是把 `$(npm prefix -g)/bin` 加进 `~/.bashrc` 或 `~/.zshrc`，然后重新打开终端或刷新 shell。

## 你可以这样理解

- **不是致命问题**
- **是必须修的环境变量问题**
- **不修的话，最常见后果就是新终端里找不到 `openclaw` 命令**





# 3.安装过程中概念解释

## 1) Hooks（钩子）到底是什么？

**Hooks = 事件驱动的小脚本**：当 OpenClaw 里发生某些“事件”（比如你输入 `/new`、`/reset`、网关启动等）时，Gateway 会自动触发对应脚本去做事（比如保存记忆、写日志、加载引导文件）。官方定义就是：用于“响应 agent 命令与事件的自动化系统”，类似 skills 那样可被发现、可用 CLI 管理。 

### 2) “Enable hooks?” 这几个选项分别做什么？

OpenClaw 默认提供 3 个**内置（bundled）hooks**，你的界面里正是它们： 

#### ✅ boot-md（启动引导）

- **作用**：每次 Gateway 启动时，自动运行/加载 `BOOT.md`。 
- **你能用它做什么**：把“固定规则/偏好/项目背景”写进 `BOOT.md`，这样**每次启动都自动注入**，让 agent 更稳定地按你的方式工作（比如你的安全规则、回答风格、常用路径、常用工作流说明等）。

#### ✅ command-logger（命令审计日志）

- **作用**：把所有 command 事件记录到日志文件（官方给的默认位置是 `~/.openclaw/logs/commands.log`）。 
- **适用场景**：排障、复盘、审计（你想知道它到底执行过什么）。
- **注意**：日志可能包含敏感信息（你发的指令、某些参数等），相当于“操作轨迹”。

#### ✅ session-memory（保存会话快照/记忆）

- **作用**：当你输入 `/new`（重开会话）时，把当前会话上下文保存到你的 workspace（默认 `~/.openclaw/workspace/memory/`）。 
- **体验层面的好处**：你重开会话或下次再来，能更连续、更像“有记忆”。

#### ⏭ Skip for now（先跳过）

- **作用**：这三个推荐 hook 都先不启用（以后也能用命令再开）。

### 3) 你该怎么选？（实用建议）

- **大多数个人使用（强烈建议）**：三个都勾上（boot-md + command-logger + session-memory）
   因为它们分别解决：**启动一致性 / 排障可追溯 / 上下文连续性**。 
- **你特别在意隐私或磁盘**：可以先不启用 `command-logger` 或 `session-memory`（它们会把信息落盘）。
- **在 VPS/公网环境**：更要严肃对待“日志/记忆文件”和网关权限（下面 Token 部分就是核心）。

### 4) “Start TUI (best option!) / Hatch in TUI” 是干嘛的？

#### Hatch / Start TUI 的含义

这一步是“**让你的 agent 变成你的 agent**”：你会在终端里回答一些配置/偏好问题，然后进入 **TUI（终端聊天界面）**开始对话。

TUI 官方就是一个终端 UI：连上 Gateway，然后你在终端里直接发消息、看工具调用、切 session、选模型等。 

界面里那句：

> We will send: “Wake up, my friend!”
>  意思是：首次孵化/启动时，TUI 会自动发一条“唤醒消息”来测试 agent 是否真的跑起来。官方 FAQ 也明确提到：TUI 会在首次启动时自动发送这句。 

#### “How do you want to hatch your bot?” 三个选项

- **Hatch in TUI (recommended)**：在终端完成孵化 + 直接进入终端聊天（适合 VPS/SSH、打不开网页、或者就喜欢命令行）。
- **Open the Web UI**：用浏览器的 Control UI（控制面板）来孵化/配置。
- **Do this later**：先跳过，之后再 `openclaw tui` 或 `openclaw dashboard` 继续。

#  4. Token（网关令牌）这块到底在讲什么？为什么重要？

现在新命令， 直接复制tokens

```
openclaw dashboard --no-open
```

你第二张图的 **Token** 区域是在解释：**Gateway token 是 Gateway + Control UI 共用的“管理员钥匙”**。

关键点（官方文档也写得很明确）：

- Control UI（浏览器面板）是管理入口（聊天、配置、执行审批等），**不要暴露到公网**。 
- Token 来源：`gateway.auth.token` 或环境变量 `OPENCLAW_GATEWAY_TOKEN`；UI 首次连接后会把 token 存在浏览器 `localStorage`。 
- 如果你看到你之前那个报错 **“unauthorized / gateway token missing”**，本质就是：UI 这边没带 token（或 token 不匹配），需要从服务器取出 token 粘贴进去。 
- Token 可以用 doctor 生成；网关默认是“没有配置就拒绝连接（fail-closed）”，这也是安全设计的一部分。 

常用命令（对照你图里的文字）：

```
# 查看 token（在网关所在机器上执行）
openclaw config get gateway.auth.token

# 生成/重置 token
openclaw doctor --generate-gateway-token

# 重新打开仪表板（本机或配合隧道）
openclaw dashboard
```

# 6. “审批”流程： 找不到 `device/pending.json`：这是教程过期 + 文件“只有在有待审批请求时才会出现”

你看的视频说的路径大概率是旧版/旧说法。**最新版官方文档明确：**

#### 节点设备（iOS/Android/macOS/headless）配对的 pending 存在这里：

- `~/.openclaw/devices/pending.json`
- `~/.openclaw/devices/paired.json` 

也就是说你 Windows 上对应是：
 `C:\Users\Helios\.openclaw\devices\pending.json`

**但关键点：**`pending.json` 只有在“真的出现一个待审批设备请求”时才会生成，而且 pending 会过期
 所以你现在没有 `devices` 文件夹，是完全正常的：**说明目前还没有任何待配对设备请求写入。**

#### 你应该用 CLI 审批，而不是手改 JSON

官方给的审批方式是：

```
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

# 补充说明：

## 1. 配对说明：新设备接入tailscale的方式

```
openclaw devices list
openclaw devices approve <requestId>
```

### (1) `Proxy headers detected from untrusted address… Configure gateway.trustedProxies …` —— 这是 Tailscale Serve 作为反代引入的“信任代理”配置

Tailscale Serve 会以“反向代理”的方式把 HTTPS 流量转到你本机 loopback 的 gateway，并带上转发头（X-Forwarded-*）。OpenClaw 看到这些代理头但不信任来源时，就会警告，并且**不会把它当作本地连接**。

官方在 “使用 Tailscale Serve” 的平台示例里明确写了需要加：`gateway.trustedProxies '["127.0.0.1"]'`。

**按官方推荐修复（强烈建议做）**：

```
openclaw config set gateway.trustedProxies '["127.0.0.1","::1"]'
openclaw gateway restart
```

（只配 `["127.0.0.1"]` 也行；我加 `::1` 是兼容 IPv6 loopback。）



### （2）使用tailscale

Tailscale 最推荐的方式是：**OpenClaw 仍只绑定本机回环（127.0.0.1），由 Tailscale Serve 在 Tailnet 内提供 HTTPS 访问**。OpenClaw 官方也把这作为“Tailnet-only（Serve）”的优先方案。

------

#### 1) 推荐方案（强烈建议）：OpenClaw + Tailscale Serve（Tailnet 内 HTTPS）

##### A. Tailnet 侧一次性准备（非常关键）

Tailscale Serve 需要你在 tailnet 里启用 **HTTPS 证书**，否则会反复弹确认/用不了。官方说明：Serve 会提示你去做启用。

> 你可以直接先跑一次 `tailscale serve`，如果条件不满足，它会给你一个网页链接引导你在 Tailscale 管理后台开启 HTTPS。

授权一下

```
tailscale serve --bg --yes 18789
```

（可选但很建议）在 Tailscale 后台启用 **MagicDNS**，这样你能用 `xxx.ts.net` 这种名字访问而不是记 IP。

------

##### B. Windows 本机检查（确认 Tailscale 正常）

用管理员 PowerShell 跑：

```
tailscale status tailscale ip -4
```

- `status` 能看到你的设备在线、同一 tailnet 里的其它设备也在列表里
- `ip -4` 会给你本机的 100.x Tailscale IP

------

##### C. OpenClaw 一键启用 Serve（最省心）

OpenClaw 官方提供了直接命令：

```
openclaw gateway --tailscale serve
```

这会保持 Gateway **仍绑定在 127.0.0.1**，同时让 Tailscale Serve 在 tailnet 内提供 HTTPS + 路由（并且 Serve 会注入 identity headers）。

或者你也可以写配置（等效）：

```
{   gateway: {     bind: "loopback",     tailscale: { mode: "serve" },   }, }
```

## 2. 解决可能存在端口占用问题

```
ss -tulpn | grep 18789
```

## 查找进程树找到托管和服务

```
pstree -sp 88891
```

## 一、先精准找出它属于哪个 service

### 方法 1：最直接，查 cgroup

请执行：

```
cat /proc/88880/cgroup
```

你大概率会看到类似下面其中一种：

#### 情况 A：系统级 service

```
0::/system.slice/openproduct-frontend.service
```

这说明它属于：

```
openproduct-frontend.service
```

------

#### 情况 B：用户级 service

```
0::/user.slice/user-1000.slice/user@1000.service/app.slice/openproduct-frontend.service
```

这说明它属于 **用户级 systemd service**，也是：

```
openproduct-frontend.service
```

但要用 `systemctl --user` 管。

## 二、正确停止它

------

### 情况 A：系统级 service

如果查出来是系统级的，就执行：

```
sudo systemctl stop openproduct-frontend.service
sudo systemctl disable openproduct-frontend.service
```

如果你想**彻底防止任何方式再次启动**：

```
sudo systemctl mask openproduct-frontend.service
```

然后检查端口：

```
sudo ss -tulpn | grep ':3000'
```

------

### 情况 B：用户级 service

如果查出来是用户级的，就执行：

```
systemctl --user stop openproduct-frontend.service
systemctl --user disable openproduct-frontend.service
```

如果还想彻底禁止：

```
systemctl --user mask openproduct-frontend.service
```

再检查端口：

```
sudo ss -tulpn | grep ':3000'
```
